## 1. 整体架构设计 (System Architecture)

本方案采用"**设计态配置化、运行态轻量化**"的思路，核心逻辑完全下沉至客户端。

### 1.1 架构分层

* **数据层 (Data)**：Schema 字典管理、Mock 数据管理
* **配置层 (Configuration)**：可视化设计器产生 JSON 格式的模板描述文件
* **服务层 (Service)**：Express + TypeScript RESTful API
* **执行层 (Execution)**：SDK 载入模板与实时数据，完成数据加工与虚拟分页

### 1.2 模块化架构（SDK 与 Designer 分离）

**SDK（核心逻辑层）**
- 打印引擎 (PrintEngine)
- 组件渲染器 (Renderers)
- 数据管道执行器 (Pipe Executors)
- HTML 模板生成 (HTMLTemplate)
- 类型定义 (Types)
- 使用 npm link 进行本地集成

**Designer（UI 交互层）**
- 可视化设计器 (Canvas, AssetPanel, PropertyPanel)
- 组件库 (ComponentLibrary)
- 数据管道配置器 (Pipe Configurators) - 只负责 UI 配置
- 打印预览 (PrintPreview) - 调用 SDK 生成 HTML
- 状态管理 (Zustand Store)

**模块分离原则**
- SDK 只包含纯逻辑，无 UI 依赖（无 React, antd）
- Designer 从 SDK 导入所有打印相关功能，不重复实现
- 管道系统：执行器在 SDK，配置器在 Designer
- 新增功能优先在 SDK 实现，Designer 调用

### 1.3 技术栈

**前端设计器**
- React 18 + TypeScript
- Vite (构建工具)
- Zustand (状态管理)
- Ant Design (组件库)
- Monaco Editor (JSON 编辑器)

**后端服务**
- Node.js + Express
- TypeScript
- 内存存储（开发阶段）

**打印 SDK**
- TypeScript
- HTML/CSS 渲染引擎
- 浏览器原生打印 API
- **完全解耦设计**：
  - 无需配置（无 config）
  - 无需初始化（无 init）
  - 无状态管理（无缓存）
  - 数据驱动（直接接收模板和数据）

## 2. 组件映射交互流程设计 (Component Interaction Flow)

这是本方案的核心重点，描述了数据如何从“资产树”经过“管道”最终绑定到“画布组件”。

### 2.1 交互时序流程图

该流程展示了用户从拖拽字段到预览生效的完整闭环。

1. **解析资产树**：设计器解析业务 Schema，映射为带有中文 Label 的树形结构。
2. **拖拽绑定事件**：用户执行拖拽，系统记录 `TargetComponent.binding.path = DraggedField.key`。
3. **管道注入 (Pipes)**：用户在属性面板为绑定字段选择 `Formatter`。
4. **实时解构预览**：系统监听绑定变化，立即调用 `Path Resolver` 取出模拟数据，通过 `Pipe` 函数序列，将结果更新至画布 DOM。

### 2.2 数据绑定模型 (Binding Data Model)

每个组件的映射关系由以下逻辑结构支撑：

* **Path Resolver**：支持点号路径（如 `a.b.c`）的安全取值。
* **Scope Manager**：在表格等循环组件内，自动将根路径转换为相对路径。
* **Pipe Executor**：按数组顺序执行格式化函数链。

### 2.3 管道系统架构（插件化设计）

- 采用注册器模式 (Registry Pattern)
- 执行器与配置器分离 (Executor/Configurator Separation)
- 新增管道只需 3 步：创建执行器 → 创建配置器 → 注册

## 3. 核心功能实现方案 (Key Implementations)

### 3.1 混合渲染引擎 (Hybrid Rendering Engine)

#### 3.1.1 架构演进

**V1.0 - 虚拟分页（已废弃）**
- 问题：组件保持原始 yMm，表格后组件位置错误
- 问题：无法支持图层（水印、背景图）

**V1.1 - 原生分页（临时方案）**
- 优势：简单、快速
- 问题：仍然无法解决组件重叠、不支持图层

*V1.4 - 表格前后分组方案（已废弃）**
- 问题：需要区分表格前后组件，逻辑复杂
- 问题：不易扩展多表格场景

**V1.5 - 相对间距 (gap) 模型（当前方案）** ✅
- 核心思想：所有组件按相对间距处理
- 解决：逻辑统一、易于理解、天然支持多表格
- 支持：紧凑布局、组件重叠 (gap < 0)
- 优势：符合"间距"思维直觉

#### 3.1.2 核心原理（相对间距模型）

**设计思想**

采用"相对间距 + 统一遍历"模型：

1. **预处理阶段**：计算每个组件与上一个组件底部的间距 (gap)
2. **渲染阶段**：统一遍历，累加高度，遇到表格就拆分
3. **换页规则**：新页面第一个组件从 marginTop 开始，忽略原 gap

**相对间距计算**

```typescript
// 1. 按 yMm 排序
const sorted = components.sort((a, b) => a.yMm - b.yMm);

// 2. 计算相对间距
interface ComponentWithGap {
  comp: ComponentNode;
  gap: number;  // 与上一个组件底部的间距 (mm)
}

const componentsWithGaps = sorted.map((comp, index) => {
  if (index === 0) {
    // 第一个组件：间距 = 距离页面顶部
    return { comp, gap: comp.yMm };
  }
  
  const prevComp = sorted[index - 1];
  const prevBottom = prevComp.yMm + prevComp.heightMm;
  const gap = comp.yMm - prevBottom;  // 可以为负（重叠）
  
  return { comp, gap };
});
```

**统一遍历处理**

```typescript
let currentPageHeight = marginTop;
let isFirstComponentInPage = true;

for (const { comp, gap } of componentsWithGaps) {
  const compHeightMm = comp.heightMm || 50;
  
  if (comp.type === 'table') {
    // 表格：跨页拆分
    const result = splitTableWithGap(
      comp, tableData, gap, isFirstComponentInPage, ...
    );
    currentPageHeight = result.currentPageHeight;
    isFirstComponentInPage = false;
  } else {
    // 普通组件：累加高度
    const needHeight = (isFirstComponentInPage ? 0 : gap) + compHeightMm;
    
    if (currentPageHeight + needHeight > availableHeight) {
      // 换页
      pages.push(currentPage);
      currentPage = [];
      currentPageHeight = marginTop;
      isFirstComponentInPage = true;  // 新页第一个组件
    }
    
    if (isFirstComponentInPage) {
      comp.yMm = currentPageHeight;  // 忽略 gap
      currentPageHeight += compHeightMm;
    } else {
      currentPageHeight += gap;  // 应用 gap
      comp.yMm = currentPageHeight;
      currentPageHeight += compHeightMm;
    }
    
    currentPage.push(comp);
    isFirstComponentInPage = false;
  }
}
```

**表格跨页拆分（支持 gap 参数）**

```typescript
function splitTableWithGap(
  tableComponent: ComponentNode,
  tableData: any[],
  gap: number,  // 表格与上一个组件的间距
  isFirstComponentInPage: boolean,
  availableHeightMm: number,
  currentPageHeight: number,
  pages: ComponentNode[][],
  currentPage: ComponentNode[]
) {
  const headerHeight = 10;  // mm
  const rowHeight = 10.4;   // mm
  const marginTop = 5;      // mm
  
  let workingPageHeight = currentPageHeight;
  let isFirstFragment = true;
  
  while (remainingData.length > 0) {
    // 计算当前页剩余高度
    let remainingHeight = availableHeightMm - workingPageHeight;
    
    // 第一个片段需要考虑 gap
    if (isFirstFragment && !isFirstComponentInPage) {
      remainingHeight -= gap;
    }
    
    // 计算能放多少行
    const needHeader = !isFirstFragment;
    const availableForRows = remainingHeight - (needHeader ? headerHeight : 0);
    const rowsCanFit = Math.floor(availableForRows / rowHeight);
    
    if (rowsCanFit <= 0) {
      // 当前页放不下，换页
      pages.push(workingPage);
      workingPage = [];
      workingPageHeight = marginTop;  // 重置为页边距
      isFirstFragment = false;
      continue;
    }
    
    // 创建表格片段
    const tableFragmentYMm = isFirstFragment
      ? (isFirstComponentInPage ? workingPageHeight : workingPageHeight + gap)
      : marginTop;
    
    const tableFragment = {
      ...tableComponent,
      layout: { yMm: tableFragmentYMm },
      props: { _pageData: dataForThisPage }
    };
    
    workingPage.push(tableFragment);
    
    // 更新高度
    const tableFragmentHeight = headerHeight + dataForThisPage.length * rowHeight;
    if (isFirstFragment && !isFirstComponentInPage) {
      workingPageHeight += gap + tableFragmentHeight;  // 第一页应用 gap
    } else {
      workingPageHeight += tableFragmentHeight;
    }
    
    // 如果还有剩余数据，换页
    if (remainingData.length > 0) {
      pages.push(workingPage);
      workingPage = [];
      workingPageHeight = marginTop;  // 第二页从 marginTop 开始
      isFirstFragment = false;
    }
  }
  
  return {
    currentPage: workingPage,
    currentPageHeight: workingPageHeight  // 返回最后一页的高度
  };
}
```

**核心亮点**

1. **逻辑统一** ✅
   - 所有组件按相对间距处理，无需分组
   - 表格前后逻辑一致
   - 天然支持多表格场景

2. **易于扩展** ✅
   - 未来新增动态高度组件（列表、容器）
   - 无需修改核心逻辑
   - 只需实现跨页拆分方法

3. **符合直觉** ✅
   - “间距”思维模型
   - 允许负 gap（组件重叠）
   - 允许 0 gap（紧凑布局）

4. **换页规则** ✅
   - 新页面第一个组件从 marginTop 开始
   - 忽略原 gap，避免首行空白
   - 表格第二页同样从 marginTop 开始

## 3.2 二维码/条形码同步渲染方案 (QR/Barcode Synchronous Rendering)

### 3.2.1 核心策略

**设计目标**
- ✅ 彻底消除异步等待：从 CDN 加载 + 异步渲染（1-4秒）改为同步生成（<10ms）
- ✅ 简化代码逻辑：移除 144 行轮询检测代码
- ✅ 提升打印体验：无需等待，立即打印
- ✅ 离线可用：不依赖 CDN 网络连接

**技术选型**
- **二维码**：`qrcode` (支持 canvas 同步生成)
- **条形码**：`jsbarcode` (支持 canvas 同步生成)
- **输出格式**：base64 编码的 PNG 图片，内嵌在 `<img>` 标签中

### 3.2.2 实现原理

**二维码渲染器 (QRCodeRenderer.ts)**

```typescript
import QRCode from 'qrcode';

export class QRCodeRenderer implements ComponentRenderer {
  render(component: ComponentNode, context: RenderContext): string {
    const content = context.resolveBinding(component.binding) || component.props?.content;
    const sizePx = component.layout.widthMm * context.mmToPx;
    
    try {
      // 1. 创建临时 canvas（不插入 DOM）
      const canvas = document.createElement('canvas');
      canvas.width = sizePx;
      canvas.height = sizePx;
      
      // 2. 同步生成二维码到 canvas
      QRCode.toCanvas(canvas, content, { 
        width: sizePx,
        margin: 0,
        errorCorrectionLevel: 'M'
      }, (error) => {
        if (error) console.error('QRCode generation error:', error);
      });
      
      // 3. 立即转换为 base64（同步操作）
      const dataUrl = canvas.toDataURL('image/png');
      
      // 4. 返回 img 标签
      return `<img src="${dataUrl}" alt="QR Code" style="${styleStr} display: block;" />`;
    } catch (error) {
      // 失败时返回占位图
      return `<div style="${styleStr} border: 1px dashed #ccc;">QR Error</div>`;
    }
  }
}
```

**条形码渲染器 (BarcodeRenderer.ts)**

```typescript
import JsBarcode from 'jsbarcode';

export class BarcodeRenderer implements ComponentRenderer {
  render(component: ComponentNode, context: RenderContext): string {
    const content = context.resolveBinding(component.binding) || component.props?.content;
    const format = component.props?.format || 'CODE128';
    const heightPx = component.layout.heightMm * context.mmToPx;
    
    try {
      // 1. 创建临时 canvas
      const canvas = document.createElement('canvas');
      
      // 2. 使用 JsBarcode 生成条形码（同步）
      JsBarcode(canvas, content, {
        format: format,
        width: 2,
        height: heightPx * 0.6,
        displayValue: false,
        margin: 0,
      });
      
      // 3. 立即转换为 base64
      const dataUrl = canvas.toDataURL('image/png');
      
      // 4. 返回 img 标签
      return `<img src="${dataUrl}" alt="Barcode" style="${styleStr} display: block;" />`;
    } catch (error) {
      return `<div style="${styleStr} border: 1px dashed #ccc;">Barcode Error</div>`;
    }
  }
}
```

### 3.2.3 渲染流程

**之前（异步方案）**

```
1. HTML 中引入 CDN 脚本
   <script src="https://cdn.../qrcode.min.js"></script>
   <script src="https://cdn.../jsbarcode.min.js"></script>

2. 输出 canvas 标签
   <canvas data-qr-content="xxx"></canvas>

3. 页面加载后执行异步渲染
   - waitForLibraries() // 轮询检测 CDN 加载（0.5-3s）
   - renderQRCodesAndBarcodes() // 异步生成（0.1-0.3s）
   - 设置 data-qr-barcode-ready='true' 标记

4. 打印前等待渲染完成
   - checkReady() // 轮询检测标记（0.3-0.5s）
   - printWindow.print() // 执行打印

总耗时：1.5-4秒
```

**现在（同步方案）**

```
1. 渲染器同步生成 base64
   const canvas = document.createElement('canvas');
```

## 3.3 布局模式实现

#### 3.2.1 基础布局模式

* **绝对定位 (Absolute)**：CSS `position: absolute`，单位固定为 `mm`。
* **流式排版 (Flow)**：基于 DOM 正常流，组件之间存在垂直方向的堆叠关系（Block Flow）。当上方动态组件高度增加时，通过 `relative` 偏移量或重绘下方的坐标，实现自动下移。

#### 3.2.2 块级组件（Block-Level Components）

**概念定义**

块级组件是指具有动态高度特性、需要占据整行宽度的组件类型。目前包括：
- **表格（Table）**：根据数据行数动态计算高度
- 未来可能扩展：列表、容器等

**核心特性**

1. **自动宽度**
   - 打印渲染时自动占满可用宽度
   - 可用宽度 = 页面宽度 - 左右边距
   - 设计器中可手动调整（仅用于预览，打印时会被覆盖）

2. **相对间距布局**
   - 设计器：保持绝对定位，用户自由拖拽
   - 打印时：计算相对间距，自动跟随上一个组件
   - 换页时：从 marginTop 开始，忽略原间距

3. **实现原理**
   ```typescript
   // 打印渲染时的相对间距布局
   calculatePages(components) {
     // 1. 计算所有组件的相对间距
     const componentsWithGaps = components.map((comp, i) => {
       if (i === 0) return { comp, gap: comp.yMm };
       const prev = components[i - 1];
       const gap = comp.yMm - (prev.yMm + prev.heightMm);
       return { comp, gap };
     });
     
     // 2. 统一遍历，累加高度
     for (const { comp, gap } of componentsWithGaps) {
       if (comp.type === 'table') {
         splitTableWithGap(comp, gap, ...);
       } else {
         currentPageHeight += gap + comp.heightMm;
       }
     }
   }
   ```

***用户体验**

```
设计时（设计器）:
┌─────────────────────┐
│ 标题                │ y: 10mm
├─────────────────────┤
│ 表格（固定 50mm）   │ y: 30mm, gap: 20mm
├─────────────────────┤
│ 总计：XXX           │ y: 85mm, gap: 5mm
└─────────────────────┘

打印时（自动调整）:
┌─────────────────────┐
│ 标题                │ y: 10mm
├─────────────────────┤
│ 表格（实际 200mm）  │ y: 30mm（保持 20mm gap）
│   30 行数据         │
├─────────────────────┤
│ 总计：XXX           │ y: 235mm（保持 5mm gap）
└─────────────────────┘
```

**优势**
- ✅ 零学习成本：用户按视觉顺序放置组件
- ✅ 自然直觉：所有组件按间距关系自动跟随
- ✅ 自动宽度：块级组件自动占满可用宽度
- ✅ 避免重叠：相对间距保持一致

## 6. 技术性能与稳定性保障

* **渲染防抖**：在设计器中，数据绑定引起的重绘采用 `Debounce` 处理，确保输入 Pipe 参数时画布不卡顿。
* **内存管理**：SDK 在执行大批量（100页+）预览时，定期清理不再显示的虚拟 DOM 节点，防止内存溢出。
* **异常隔离**：任何数据转换错误（如对字符串应用金额格式化）均需触发回退逻辑，输出原始字段内容并标记 UI 警告。
