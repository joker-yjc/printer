## 通用打印服务平台 - MVP 实施规划

### 1. 文档说明

- **目标**：规划通用打印服务平台的 **MVP 实施路线**，并通过勾选项（`[ ] / [x]`）跟踪每个阶段的完成情况。  
- **范围**：覆盖 **模板/数据模型设计、模板管理服务、React 设计器、核心 SDK & 虚拟分页、端到端 Demo** 五个阶段。  
- **使用方式**：每完成一项，把 `- [ ]` 改成 `- [x]`，即可直观看进度。

---

### 2. 阶段一：模型与接口设计

**阶段目标**：确定前后端共享的“语言”和边界，包括模板 JSON、Schema 字典和 SDK API，是后续所有工作的地基。

- [ ] 定义模板 JSON 结构（页面、组件、布局、数据绑定、Pipe 配置）
- [ ] 定义 Schema 字典结构（字段 `key`、`label`、`type`、`enum` 等）
- [ ] 设计 SDK 对外 API（如 `PrintSDK.render` 的参数和返回约定）
- [ ] 用一两个简单示例（如“发货单”、“对账单”）验证模板结构的可表达性

**阶段完成标准**：  
- 有一份稳定的模板 JSON 示例 + Schema 示例；  
- SDK 的核心 API 形式固定下来，后续实现基本不再大改函数签名。

---

### 3. 阶段二：模板管理服务（Node + Express）

**阶段目标**：提供模板、Schema、Mock 数据的持久化能力，支撑设计器和 SDK 的加载/保存。

- [ ] 初始化 Node + TypeScript + Express 项目骨架
- [ ] 设计并实现 Schema 管理接口（`/schemas` CRUD）
- [ ] 设计并实现模板管理接口（`/templates` CRUD）
- [ ] 设计并实现模拟数据管理接口（`/mock-data` CRUD）
- [ ] 为上述接口补充基础校验和错误处理（如必填字段、ID 不存在等）
- [ ] 提供简单的环境配置（端口、跨域 CORS 等）

**阶段完成标准**：  
- 本地可启动服务；  
- 用 Postman/浏览器能完成模板/Schema/Mock 数据的新增、查询、修改、删除；  
- 错误输入时能返回清晰的错误信息。

---

### 4. 阶段三：React 设计器骨架

**阶段目标**：搭出“能用”的设计器框架，完成资产树 → 画布 → 绑定面板的基本闭环。

- [ ] 初始化 React + Vite + TypeScript 项目
- [ ] 搭建三栏布局：**左（资产树）/ 中（标尺画布）/ 右（绑定与属性）**
- [ ] 实现基于 Schema JSON 的资产树展示（支持字段类型标识：String/Number/Array 等）
- [ ] 实现画布基础渲染与标尺（单位 `mm`，简单网格即可）
- [ ] 实现基础组件（文本 / 图片 / 矩形）的拖拽到画布和选中高亮
- [ ] 实现右侧绑定信息展示（当前选中组件的数据路径 + Pipe 简单配置）
- [ ] 打通设计器与模板管理服务（模板的保存 / 加载）
- [ ] 对关键交互（拖拽、属性修改）增加简单防抖，保证编辑时不卡顿

**阶段完成标准**：  
- 能从某个 Schema 生成资产树；  
- 用户可以拖字段到画布，配置绑定，并把结果作为模板保存到后端，再次打开可以恢复。

---

### 5. 阶段四：核心 SDK 与虚拟分页 MVP

**阶段目标**：实现无框架的核心引擎 + 最小可用的分页逻辑，并提供给业务前端集成的 SDK API。

- [ ] 抽象并实现 `core-engine`（数据绑定 + 布局描述 + 测量接口）
- [ ] 在隐藏容器中渲染组件并通过 `getBoundingClientRect()` 测量高度
- [ ] 实现基础虚拟分页逻辑（按页高拆分、流式组件整体换页）
- [ ] 实现基础防断裂策略（组件超出当前页剩余高度时整体移到下一页，不做中间切割）
- [ ] 提供 SDK 对外 API（例如 `PrintSDK.render({ container, templateId, data })`）
- [ ] 在 Demo 页面中集成 SDK，完成预览区域渲染和 `window.print()` 调用
- [ ] 在 Chrome/Edge（Win/Mac）验证 100 页以内的渲染性能和打印效果

**阶段完成标准**：  
- 不依赖 React/Vue 的 SDK 打包产物可被 Demo 页面直接使用；  
- 在受控数据量下（例如 50~100 页）预览时间和分页表现满足 PRD 中的性能预期。

---

### 6. 阶段五：端到端 Demo 与回顾

**阶段目标**：用一个真实/接近真实的业务场景，从数据到打印跑通全链路，并基于体验做第一轮回顾。

- [ ] 选取一个真实业务场景（如电商发货单或对账单）准备示例数据和 Schema
- [ ] 在设计器中完成该场景的模板设计并保存到模板管理服务
- [ ] 在业务 Demo 页面中通过 SDK 加载模板 + 实际/模拟数据，完成预览与打印
- [ ] 联合业务视角评估：  
  - 布局能力是否满足主需求  
  - 分页和表头跨页是否符合预期  
  - 性能是否 OK（是否有明显卡顿）
- [ ] 梳理下一阶段优化清单（如：更多组件类型、表格高级能力、更细的权限/多租户等）

**阶段完成标准**：  
- 至少有一个“可以给内部观众演示”的完整场景 Demo；  
- 输出一份问题和优化清单，作为 V1.1 / V2 的输入。
